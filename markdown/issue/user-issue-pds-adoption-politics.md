# PDS Adoption - Internal Resistance vs External Validation

PDS (vendor-agnostic data platform) successfully adopted by other UO departments and external schools through Brock's code sharing and advising, but blocked from broader adoption within own department (UO Information Services). Labeled "controversial" and restricted to personal use only.

## Current Situation (2025-11-22)

**External validation:**
- Production use: Other UO departments
- Production use: Other schools
- Achieved through Brock's direct advising and code sharing

**Internal constraint:**
- Status: "Controversial"
- Allowed users: Brock and Sameer (Data Engineer) - personal use only
- Restriction: Not approved for broader team/departmental adoption

**Nature of resistance:** Political/institutional, not technical

## Organizational Context

**Team structure:**
- Division: Enterprise Solutions (Associate CIO position currently vacant)
- Team: ~5-6 people (Brock, Sameer, Jody, Rachael, Tim, reporting to Travis)
- Brock's role: ETL Developer, specialized role within team

**Stakeholder landscape (identified but not yet detailed):**
- Travis (boss/manager) - role in resistance unknown
- Vacant Associate CIO - leadership vacuum impact unknown
- DBAs - potential Oracle turf protection unknown
- Team members - investment in current stack unknown

## Strategic Response

**Brock's navigation approach:**

- External advocacy: Share code and advise other institutions
- Constraint routing: Build future anyway within personal use allowance

**"Prophet in own land" dynamics:**

- Institutional antibodies internally
- Common in risk-averse enterprise environments

**Dual infrastructure as routing strategy:**
- Not confronting constraints directly
- Building future while maintaining present
